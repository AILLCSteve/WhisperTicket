## 1. High-Level Summary

WaitTicket (repo: `Whisper/`, Xcode target: `WhisperTicket`) is an iOS app for restaurant waitstaff that turns spoken orders into structured, editable ticket records. Core loop: server selects a table on a visual floor plan, holds a mic button and speaks the order, live on-device speech-to-text builds a structured draft in real time (parsing items, seats, courses, modifiers, allergies), the draft is reviewed / confirmed / manually edited, and sent to kitchen.

The app is **fully on-device** — no backend required. Every service (audio, ASR, menu store, order parser, upsell engine, ticket repository, floor plan store) is abstracted behind a protocol and injected at app startup. Swapping concrete implementations in `WhisperTicketApp.swift` is the only change needed for Phase 2 (Supabase backend).

**Implemented feature tiers:** Table select grid + floor plan canvas, hold-to-talk ASR, fuzzy menu order parsing, allergy guardrails, repeat-back coaching sheet, noise detection warning, auto abbreviations, rule-based upsell engine with playbook scripts, seat map (drag-and-drop), course pacing (Fire / Hold per course), kitchen note templates, voice macros, full ticket edit + audit trail, CI via GitHub Actions → TestFlight. **No third-party dependencies.** iOS 17.0 minimum. Swift 5.9. Xcode project generated by `xcodegen`.

### 4.2 TranscriptionServiceProtocol → SFSpeechTranscriptionService
**File:** `Services/SFSpeechTranscriptionService.swift`

```swift
protocol TranscriptionServiceProtocol: AnyObject {
    func transcriptionPublisher() -> AnyPublisher<TranscriptionSegment, Never>
    func startTranscribing(audioPublisher: AnyPublisher<AVAudioPCMBuffer, Never>) throws
    func stopTranscribing()
}

struct TranscriptionSegment { let text: String; let isFinal: Bool }
```

**Implementation details:**
- `SFSpeechRecognizer(locale: Locale(identifier: "en-US"))`, `requiresOnDeviceRecognition = true`
- **Auto-restart on final result:** When `result.isFinal = true`, accumulates text in `accumulatedBase`, cancels task, creates new `SFSpeechAudioBufferRecognitionRequest`, restarts recognition task with same audio publisher. Maintains continuity across iOS 60-second ASR limit.
- `stopTranscribing()` sets `isSessionActive = false` → halts auto-restart loop
- Published via `PassthroughSubject<TranscriptionSegment, Never>`
- `static func requestPermission(completion: (Bool) -> Void)` — `SFSpeechRecognizer.requestAuthorization`
- Errors during active session are logged but swallowed; only explicit stop triggers error propagation

### 4.3 OrderParserProtocol → FuzzyMenuOrderParser
**File:** `Services/FuzzyMenuOrderParser.swift`

```swift
protocol OrderParserProtocol {
    func parseDraft(transcript: String, existingDraft: TicketDraft, menu: MenuV1) -> TicketDraft
    func detectMacro(in text: String) -> VoiceMacro?
    func repeatBackSummary(for draft: TicketDraft) -> String
}
```

**Parse pipeline — `parseDraft(transcript:existingDraft:menu:)`:**
1. Extract `newText = transcript[consumedCursor...]`
2. Normalize: lowercase, strip filler words ("um", "uh", "like", "so", "and", "the", "a", "an", "for"), convert number-words to digits (1–10)
3. Split into segments on commas + periods
4. Per segment:
   - Match course keyword → update `currentCourse` (CourseFlag)
   - Match `seat\s+(\d+)` → update `currentSeat` (Int?)
   - `tokenOverlapScore(query, itemName)` for each MenuItem → take best match ≥ 0.4
   - Extract leading quantity (digit or number-word prefix)
   - Extract modifiers (temperature map, modifier group options) with negation prefix detection
   - Detect allergy keywords → `hasAllergyFlag = true`
   - Build `DraftItem`, call `draft.addItem(item)` (dedup enforced)
5. `consumedCursor = transcript.count` → next parse starts here

**Token overlap scoring — `tokenOverlapScore(_ a: String, _ b: String) -> Double`:**
- Normalize both strings to lowercase, split on whitespace
- Filter tokens with `count > 2`
- `score = |intersection| / |queryTokens|`
- Item match threshold: **0.4**; modifier match threshold: **0.5**

**Built-in lookup tables:**
| Table | Contents |
|-------|----------|
| `allergyKeywords` | "allergy", "allergic", "anaphylactic", "epipen", "cannot eat" |
| `fillerWords` | "um", "uh", "like", "so", "and", "the", "a", "an", "for" |
| `temperatureMap` | 8 entries: rare → "Rare", medium rare → "Medium Rare", medium well → "Medium Well", well done → "Well Done", variants |
| `numberWords` | 1–10 |
| `negationPrefixes` | "no", "without", "hold", "remove", "skip" |
| `courseKeywords` | 20+ entries → CourseFlag mapping |
| `macroPatterns` | 7 entries → VoiceMacro |

**`detectMacro(in:)`:** Normalizes text, iterates macroPatterns, returns first `VoiceMacro` match or nil.

**`repeatBackSummary(for:)`:** Builds human-readable string: "Table X: Qty×Name (Mod1, Mod2); ..." — appends "⚠️ ALLERGY" for items with hasAllergyFlag.

### 4.6 UpsellEngineProtocol → RuleBasedUpsellEngine
**File:** `Services/RuleBasedUpsellEngine.swift`

```swift
protocol UpsellEngineProtocol {
    func suggestions(for draft: TicketDraft, menu: MenuV1) -> [UpsellSuggestionResult]
}

struct UpsellSuggestionResult: Identifiable {
    let id: String   // UUID
    let menuItem: MenuItem
    let reason: String
    let playbookScript: String?
}
```

**Algorithm:**
1. `hasEntree = draft.items.contains { $0.course == .entree }`
2. `hasDrink = draft.items.contains { $0.course == .beverage }`
3. For each `UpsellRule`:
   - Check `condition.hasEntree` and `condition.hasDrink` — both must match if specified
   - For each `UpsellSuggestion` in rule:
     - Tag match: `allMenuItems.filter { $0.tags.contains(suggestion.tag) }`
     - ItemId match: `menu.item(byId: suggestion.itemId)`
   - Filter out items already in draft (by menuItemId)
   - Deduplicate by `Set<String>` of menuItem.id
4. Return results

## 7. Critical Flows — Step-by-Step

### 7.1 App Startup
1. `WhisperTicketApp.init()` — build `Schema`, attempt `ModelContainer`. On failure: wipe `.sqlite` + WAL + SHM, retry. On second failure: `fatalError`.
2. `ContentView()` rendered. `WelcomeView` sheet appears.
3. `.task {}` fires: request speech permission → mic permission → `menuStore.loadMenu()`.
4. `loadMenu()` tries 5 resolution strategies; builds `itemIndex` + `searchIndex` on MainActor.
5. `FloorPlanStore.load()` restores floor plan from UserDefaults or uses `.default`.

### 7.2 Voice Order Capture (Happy Path)
1. Server taps table on `FloorView` → navigates to `TableOrderEntryView`.
2. Selects/adds seat via `SeatChip` strip.
3. Holds `HoldToTalkButton` → `LiveSessionViewModel.startRecording()`:
   - `audioCapture.startCapture()` → AVAudioEngine running, tap installed
   - `transcriptionService.startTranscribing(audioPublisher: audioCapture.audioBufferPublisher())`
4. Audio buffers flow: `AVAudioEngine` → `PassthroughSubject` → `SFSpeechRecognizer.recognitionRequest`
5. Transcription results arrive: `SFSpeechRecognizer` → `transcriptionSubject.send(TranscriptionSegment)`
6. `handleTranscriptionSegment(_:)`:
   - Updates `seatTranscripts[activeSeatNumber]`
   - Calls `parser.parseDraft(transcript: activeSeatTranscript, existingDraft: draft, menu: menuStore.menu!)`
   - Parser extracts new text after `consumedCursor`, identifies items/seats/courses/modifiers/allergies
   - Checks `detectMacro(in:)` → sets `detectedMacro` if found
   - Updates `draft.items`
   - Calls `refreshUpsells()`
7. `isFinal = true` → `SFSpeechTranscriptionService` accumulates text, auto-restarts recognition task
8. Server releases button → `stopRecording()` → 3-second `isFinalizingTranscription` timer → `finalizeTranscription()`
9. If no items parsed → create fallback DraftItem with full transcript text

### 7.3 Allergy Detection Flow
1. Parser detects allergy keyword in transcript → sets `DraftItem.hasAllergyFlag = true`
2. `handleTranscriptionSegment` adds item to `allergyItemsPendingConfirm`
3. `LiveSessionView` renders `AllergyAlertBanner` per pending item
4. Server taps Confirm → `confirmAllergyItem(_:)` removes from pending list
5. On ticket creation: `TicketItem.hasAllergyFlag = true`, `allergyConfirmed` = whatever server set
6. `TicketEditorView` shows allergy badge per item; "Confirm Allergy" button in editor

### 7.4 Ticket Creation Flow
1. Server taps Confirm in `RepeatBackSheet` → calls `repository.createTicket(from: draft, serverId: "local_server")`
2. `createTicket` logic:
   - Collect distinct `seatNumbers` from `draft.items`
   - If empty: create `GuestSeat(seatNumber: 1)` with all items
   - Else: create one `GuestSeat` per distinct seatNumber; unseated items → Seat 1
   - Per-seat transcript: `seat.rawTranscript = draft.seatTranscripts[seatNumber] ?? ""`
   - `ticket.rawTranscript = draft.aggregateTranscript`
   - Each `DraftItem` → `buildTicketItem()` → `TicketItem` + `TicketModifier`s
3. `modelContext.insert(ticket)`, `save()`
4. Navigation pushes to `TicketEditorView(ticket: createdTicket)`

### 7.5 Course Pacing Flow
1. `TicketEditorView` shows Course Pacing section per active `CourseFlag`
2. Server taps "Fire Entrees": `vm.fireCourse(.entree)`
3. `TicketEditorViewModel.fireCourse(_:)`:
   - `coursePacingStates[.entree] = .fired`
   - `ticket.coursePacingStates["ENT"] = "FIRED"`
   - `logEdit(type: "status_changed", summary: "Fired Entrees")`
   - `save()`
4. UI reflects: color/icon changes per `CoursePacingState`
5. Reverse: "Hold Entrees" → `.holding`

### 7.6 Seat Item Drag-and-Drop
1. `TicketEditorView` → Seat Map button → `SeatMapView` sheet
2. Server drags `SeatCard` item: payload = `"itemId|sourceSeatNumber"` (String Transferable)
3. Drops on target seat card: `dropDestination` fires with payload
4. Parse payload: extract `itemId` and `sourceSeatNumber`
5. `vm.moveItem(item, fromSeat: sourceGuestSeat, toSeatNumber: targetSeatNumber)`
6. `removeItem` from source `GuestSeat.items`, append to target (create target `GuestSeat` if missing)
7. `logEdit(type: "item_moved", ...)`, `save()`

### 7.7 Voice Macro Execution
1. Parser `detectMacro(in:)` matches text → `detectedMacro` set
2. `LiveSessionView` shows macro prompt banner
3. Server taps Execute → `vm.applyMacro(macro, previousDraft: previousDraft)`
4. `.repeatLastOrder` → copy all items from `previousDraft` into `draft`
5. `.addSideSalad` → `menuStore.findBestMatches(text: "Side Salad", maxResults: 1)` → create DraftItem
6. `.splitCheck` → no-op (Phase 2)

## 9. Risks, Gaps & Open Questions

### 9.1 Confirmed Risks [DIRECTLY SUPPORTED]
| Risk | Evidence | Severity |
|------|----------|---------|
| Parser accuracy degrades at noise threshold | tokenOverlapScore 0.4 threshold empirically set; no feedback loop | Medium |
| No validation of `menuItemId` foreign key integrity | DraftItem.menuItemId = "manual_XXXX" for manual adds; no referential check | Low |
| `coursePacingStates` stored as `[String: String]` (not typed) | Requires string casting on read; desync possible if CourseFlag rawValues change | Medium |
| SwiftData migration strategy is dev-only wipe | Production would lose data; `VersionedSchema` + `MigrationPlan` not yet implemented | High (for production) |
| `requiresOnDeviceRecognition = true` — may fail on devices without downloaded model | No user-facing error path if on-device model unavailable | Medium |
| `FloorPlanStore` not protocol-abstracted | Cannot be swapped for testing or backend; only concrete class used | Low |
| No pricing calculation | `TicketModifier.priceDelta` stored but never summed into a ticket total | Low |
| `TicketEditEvent.editHistory` not shown in `TicketsListView` | History only visible per-ticket in editor | Low |

### 9.2 Inferred Gaps
- No server authentication — `serverId = "local_server"` hardcoded
- No shift management — no concept of shift start/end, assigned sections per shift
- No conflict resolution for same-table concurrent edits (deferred to Supabase Realtime)
- `RepeatBackSheet` manual item entry creates DraftItem but no menu matching (raw name stored)
- `SeatMapView` drag-and-drop uses String payload — no type safety in transfer protocol

### 9.3 Open Questions
- When is `TicketDraft.rawTranscript` vs `TicketDraft.aggregateTranscript` used? (`rawTranscript` appears unused; `aggregateTranscript` (combined seatTranscripts) is stored on Ticket)
- Should upsell deduplication be per-session (cleared on table select) or persistent?
- Is `TableOrderEntryView` replacing `LiveSessionView` or are both active navigation paths?
- `VoiceMacro.splitCheck` — what exactly splits? (Items to separate GuestSeats? Separate tickets?)

## 10. Edge Cases & Failure Modes

| Scenario | Behavior | Correct? |
|----------|---------|---------|
| Microphone permission denied | `startCapture()` throws; `errorMessage` set in VM | Partially — no recovery path shown |
| Speech permission denied | `startTranscribing()` throws; session logs warning | Partially — no UI error state |
| Menu JSON missing from bundle | `loadMenu()` throws after 5 resolution attempts with diagnostic log | Yes — extensive diagnostics |
| SwiftData migration failure | Wipe + retry in `WhisperTicketApp.init()` | Yes (dev) — No for production |
| ASR 60-second limit | Auto-restart accumulates `accumulatedBase` — continuity maintained | Yes |
| No items parsed from transcript | `finalizeTranscription()` creates fallback DraftItem with raw transcript | Yes |
| Duplicate item in same draft | `TicketDraft.addItem()` deduplicates by menuItemId + modifierNames + seatNumber | Yes |
| Seat number 0 in `TicketEditEvent` | Denotes table-level event (not a real seat) | Yes — documented convention |
| HTTP 409 on ASC compliance PATCH | Expected — caught and logged as non-fatal | Yes |
| `coursePacingStates` key not found | Returns `.holding` as default via `?? .holding` on dictionary access | Yes |
