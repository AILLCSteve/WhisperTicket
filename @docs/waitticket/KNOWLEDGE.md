## Domain Model

### Core Entities (SwiftData @Model)

**Ticket** — root aggregate for one table's ordering session.
- `tableId: String`, `tableName: String`, `status: TicketStatus` (open/sent/delivered/closed)
- `rawTranscript: String` — full accumulated speech text
- `createdAt: Date`, `sentAt: Date?`, `deliveredAt: Date?`, `closedAt: Date?`
- Relationship: `seats: [GuestSeat]` (no cascade rule set — orphans accumulate on delete)
- Relationship: `editHistory: [TicketEditEvent]`

**GuestSeat** — one guest's portion of the order.
- `seatNumber: Int` (1-indexed), `label: String` (seat label or "Seat N")
- `transcript: String` — the speech text assigned to this seat
- Relationship: `items: [TicketItem]` (no cascade rule)

**TicketItem** — one ordered menu item.
- `menuItemId: String`, `name: String`, `price: Double`, `quantity: Int`
- `course: CourseFlag` (appetizer/entree/dessert/beverage/side)
- `allergyFlagged: Bool`, `kitchenNote: String?`
- Relationship: `modifiers: [TicketModifier]`

**TicketModifier** — one modifier on an item.
- `name: String` (e.g., "No Ranch", "Well Done")
- `isNegation: Bool` — true for explicit "no X" modifiers

**TicketEditEvent** — append-only audit trail entry.
- `timestamp: Date`, `description: String`, `changeType: EditEventType`

### Floor Plan (UserDefaults, NOT SwiftData)

**FloorPlan** (Codable) — persisted at key `whisperticket.floorplan.v2`.
- `tables: [FloorTable]`, `sections: [ServerSection]`

**FloorTable** (Codable) — one restaurant table.
- `id: String` (UUID), `name: String`
- `position: CGSize` (free-floating offset in canvas, no snap)
- `seats: [SeatConfig]` (static layout seats, NOT live order seats)
- `sectionId: String?`

**SeatConfig** — static seat label within a floor plan table.
- `id: String`, `label: String`
- These are NOT `GuestSeat` — they're layout config, not order data.

### Menu (MenuV1, JSON/UserDefaults)

**MenuV1** (Codable) — loaded from bundle JSON or saved via `saveMenu()`.
- `restaurantId: String`, `version: Int`, `currency: String`
- `categories: [MenuCategory]`
- `upsellRules: [UpsellRule]`

**MenuItem** — one menu item.
- `id: String`, `name: String`, `price: Double`, `description: String`
- `tags: [String]` — taxonomy: `"beverage"`, `"dessert"`, `"entree"`, `"appetizer"`, `"side"`
- `modifierGroups: [ModifierGroup]`, `upsellLinks: [String]`
- `kitchenNoteTemplate: String?`

**UpsellRule** — one upsell trigger.
- `id: String`, `condition: UpsellCondition`, `suggest: [UpsellSuggestion]`, `playbookScript: String`

**UpsellCondition** — filter for when to suggest.
- `hasEntree: Bool?`, `hasDrink: Bool?` (nil = don't check)
- Cannot express: time-of-day, party size, item-level tag conditions, combo triggers

## Feature Map

### 1. Hold-to-Talk ASR Pipeline

**What it is:** Tap-to-start / tap-to-stop voice recording (NOT hold — despite the feature name). Microphone is a stateful toggle in `LiveMicButton`.

**Call path:**
```
LiveMicButton tap → vm.startRecording()
  → audioCapture.startCapture()
    → AVAudioSession.setCategory(.playAndRecord, mode: .measurement, options: [.defaultToSpeaker, .allowBluetoothHFP])
    → AVAudioSession.setActive(true)
    → inputNode.installTap { buffer → bufferSubject.send(buffer) }  ← RT thread
    → audioEngine.start()
  → transcriptionService.startTranscribing(audioPublisher:)
    → SFSpeechAudioBufferRecognitionRequest (shouldReportPartialResults=true, requiresOnDeviceRecognition=true)
    → recognitionTask(with: request) { result, error in ... }
    → audioCancellable: bufferPublisher.sink { request.append(buffer) }  ← RT thread
    → result handler: transcriptionSubject.send(segment)  ← Speech thread
  → transcriptionCancellable: .receive(on: .main).sink { handleTranscriptionSegment($0) }
  → handleTranscriptionSegment:
    → parseDraft() on every partial result (no debounce)
    → stamps items with activeSeatNumber
    → refreshUpsells()

Mic tap again → vm.stopRecording()
  → audioCapture.stopCapture() → removeTap → stop engine → deactivate session
  → transcriptionService.endAudioInput() → endAudio() (seal, don't cancel)
  → 1.5s finalizationTimer → finalizeTranscription()
    → stopTranscribing() (cancel task, nil refs)
    → if zero items: TranscriptCleaner.clean() → fallback DraftItem (confidence 0.5)
```

**Ecosystem position:** SFSpeechRecognizer on-device is correct choice for privacy and offline use. WhisperKit (2.2% WER, noise-robust) is the protocol-abstracted upgrade path. SpeechAnalyzer (iOS 26+) is the long-term target.

**Known risks:**
- No `AVAudioSession.interruptionNotification` handler — phone call mid-recording crashes to inconsistent state
- Buffer not copied before Combine send — engine can reuse memory before subscriber processes it
- `parseDraft()` called on every partial (~every 0.5s) with no debounce
- No `SFCustomLanguageModelData` vocabulary — restaurant menu items are poorly recognized by general model

### 2. Fuzzy Menu Order Parser

**What it is:** `FuzzyMenuOrderParser` converts accumulated speech text into structured `DraftItem` objects using token overlap scoring.

**Algorithm (exact):**
```
normalizeText(segment)
  → 9 stop words removed (separate from TranscriptCleaner's 68-phrase list)
  → number words ("one"→"1") normalized

splitIntoSegments on commas and periods
  → detectCourse / detectSeat first (consuming the sentence if matched)
  → findBestItem:
      qSet = Set(queryTokens.filter { $0.count > 2 })
      score = |qSet ∩ itemTokens| / |qSet|   ← query-side precision, NOT F1/Jaccard
      threshold: 0.4 items, 0.5 modifiers
  → extractQuantity: regex (\d+)\s+\w
  → extractModifiers: temperature map + tokenOverlapScore on modifier names
  → allergyCheck: substring scan against keyword list

finalizeTranscription fallback:
  → TranscriptCleaner.clean() (68-phrase filler removal, longest-first order required)
  → Single DraftItem at confidence 0.5 if no items parsed
```

**Known failure modes:**
- **F1 — Multi-word ambiguity**: "chicken" matches all chicken items equally; first in menu array wins
- **F8 — Compound orders**: "burger and salad" in one breath → one item only (no splitting on " and ")
- **F4 — Modifier negation bug**: creates "No Ranch" string instead of using `TicketModifier.isNegation: Bool`
- **F7 — Course detection swallows item**: "for the entree, chicken" needs the comma or item is lost
- **F9 — Default course bug**: parser defaults `currentCourse = .entree` for all items; dessert/drink upsell fires on first item regardless of type

**reparseItems():** Called from `TicketEditorViewModel` when user manually edits a transcript. Merge-only — never replaces existing items, only appends new ones. Race risk if called concurrently.

### 3. Floor Plan Editor

**What it is:** Two separate canvas systems sharing the floor plan data:
- `FloorPlanEditorView` — manager drag-and-drop table placement
- `FloorMapEmbedView` (in FloorView) — read-only operational view

**Drag mechanism:** `DragGesture()` + `@GestureState` (two-state pattern):
- Live: `@GestureState var dragDelta` (resets to .zero on lift)
- On end: commit to `@State var localPosition`, call `floorPlanStore.upsertTable(updated)` → `save()` synchronously

**Canvas issues:**
- Editor: 600×800 pt fixed canvas; embed: 900×700 pt fixed — position overflow in editor is invisible, clipped in embed
- No snap-to-grid (grid is drawn at 40pt spacing but positions are free-floating)
- No `minimumDistance` on DragGesture (default 10pt) — tiny accidental drags commit position saves
- `ScrollView` wrapping DraggableTableTile creates scroll/drag conflict — tables near edges may scroll instead of drag
- No `UndoManager` — accidental moves are permanent

**Persistence silent-reset:** `FloorPlanStore.load()` silently resets entire floor plan to default on ANY decode error. Schema change = data loss with no warning. Add decode error logging at minimum.

**SeatMapView** (per-ticket) is a completely separate feature — transfers items between `GuestSeat` objects in a live ticket using SwiftUI `.draggable`/`.dropDestination`. Not connected to FloorPlanStore.

### 4. PDF Menu Import

**What it is:** `PDFMenuImportService` uses PDFKit text extraction + heuristic regex parsing to produce a `MenuV1` from a PDF.

**Pipeline:**
```
PDFDocument(url:) → page.string concatenation → fullText
  → split on newlines, trim, drop empty
  → for each line:
      if price match ($X.XX regex) → MenuItem (price, name)
      else if isLikelyCategoryHeader (all-caps or title-case, no $, len 3-60) → flush + new category
      else → silently discard
  → defaultUpsellRules() injected
  → MenuV1 returned
```

**Critical bugs:**
- **Price regex captures integer only**: `\$?\s*(\d{1,3}(?:\.\d{2}))` — `(?:...)` is non-capturing. `$14.99` → price `14`. Fix: remove `?:` → `(\d{1,3}\.\d{2})`.
- **Tags always empty**: All imported items have `tags: []` — tag-based upsell never fires for any imported menu.

**Real-world failure modes:**
- Scanned/image PDFs: PDFKit returns empty text (correctly detected, returns error message)
- Multi-column menus: text interleaved from both columns
- Round-number prices ($15): `\.\d{2}` required — fails
- Sentence-case headers ("Pasta dishes"): `isLikelyCategoryHeader` requires all-caps or every word capitalized
- Dotted leaders (Chicken .............. 13.99): dots remain in name after price strip

**Coverage estimate:** ~30-40% of real-world restaurant PDFs (digital-native only). Scanned menus (majority of independent restaurants) require Vision framework OCR fallback.

**Phase 1.5 path:** When `fullText.isEmpty`, fall through to `VNRecognizeTextRequest` on PDFKit-rendered page CGImages. Zero new dependencies, on-device.

### 5. Rule-Based Upsell Engine

**What it is:** `RuleBasedUpsellEngine` evaluates `UpsellCondition` against the current `TicketDraft` and returns `UpsellSuggestion` candidates.

**Evaluation:**
```
hasEntree = draft.items.contains { $0.course == .entree }
hasDrink  = draft.items.contains { $0.course == .beverage }
→ for each rule in menu.upsellRules:
    if condition matches → find candidates by tag or itemId
    → deduplicate by item ID
    → return up to 2 per suggestion
```

**Known failure modes:**
- `hasEntree` is almost always true because `FuzzyMenuOrderParser` defaults `currentCourse = .entree` for all items. The dessert upsell fires immediately on the first item ordered, regardless of whether it's actually an entree.
- Tag-based suggestions are completely inert for imported menus (all `tags: []`). The engine silently returns zero candidates with no indication why.
- `defaultUpsellRules()` in PDFMenuImportService injects tag-based rules for a menu where tags are always empty — fix by using category-name tag inference or itemId-based suggestions.

**Category-name tag inference (quick fix):** After parsing, walk each `MenuCategory` and assign tags to items based on keyword matching in the category name (`"beverage"/"drink"/"bar"` → `["beverage"]`, `"dessert"/"sweet"` → `["dessert"]`, etc.).

- `TranscriptionServiceProtocol`: `startTranscribing(audioPublisher:)`, `endAudioInput()`, `stopTranscribing()`, `transcriptionPublisher()`

- `UpsellEngineProtocol`: `suggestions(for:menu:)` → `[UpsellItem]`

### MenuV1 Schema
Tag taxonomy (canonical — must match when writing upsell rules):
- `"beverage"` (NOT "drink")
- `"dessert"`, `"entree"`, `"appetizer"`, `"side"`
